## Phase 1: Database Foundation

**Goal:** Establish version-controlled database migrations, fix schema inconsistencies, and add missing infrastructure for reliable database operations.

**Requirements:** DB-01, DB-02, DB-03, DB-04, DB-05

**Success criteria:**
1. `drizzle/` directory contains an initial migration SQL file generated by `drizzle-kit generate`
2. `transactions.created_at` and `categories.created_at` are stored as `integer` with `mode: 'timestamp_ms'`
3. `idx_transactions_category` index exists on transactions.category
4. Category lazy initialization uses `onConflictDoNothing()` — no error string matching
5. Database client uses HTTP-based connection (`@libsql/client/http`) for serverless compatibility
6. Migration check passes in CI (`drizzle-kit check` exits 0)

**UI hint:** no

---

## Phase 2: Security Hardening

**Goal:** Fix critical security vulnerabilities identified in the audit — restrict CORS, harden rate limiting, sanitize LLM inputs, and validate configuration.

**Requirements:** SEC-01, SEC-02, SEC-03, SEC-04, SEC-05

**Success criteria:**
1. CORS `origin` is a dynamic function that checks against `[FRONTEND_URL, null]` — mobile auth works
2. Rate limiter returns 401 for unauthenticated LLM requests; authenticated users identified by session ID
3. User message text is sanitized (JSON stripped, delimited in prompt) before LLM ingestion
4. App startup throws descriptive error if `GITHUB_CLIENT_SECRET` or OAuth secrets are missing
5. Image input endpoint rejects non-image base64 payloads (validates `data:image/` prefix)

**UI hint:** no

---

## Phase 3: Test Infrastructure & Error Resilience

**Goal:** Make the codebase testable by introducing dependency injection for the database client, establish test environment configuration, and fix silent error swallowing in LLM calls.

**Requirements:** TST-01, TST-02, TST-03, TST-04, TST-05

**Success criteria:**
1. `DatabaseModule` registered as `@Global()` with `DRIZZLE` injection token — all 5 services use `@Inject(DRIZZLE)` instead of importing `db` singleton
2. `jest` configured with in-memory SQLite (`:memory:`) via `test/helpers/setup.ts` — `npm run test` boots without real DB
3. LLM parsing failures throw `HttpException` (502/503) instead of returning silent defaults
4. OpenRouter client has `timeout: 8000` and `maxRetries: 1`; timeout errors return clear user-facing message
5. `ConfigModule.forRoot({ validate })` checks all required env vars at startup with descriptive error messages

**UI hint:** no

## Phase 4: Automated Quality Gates

**Goal:** Write comprehensive unit and E2E tests covering all business logic, and establish CI pipeline to enforce quality on every PR.

**Requirements:** QAL-01, QAL-02, QAL-03, QAL-04

**Success criteria:**
1. Unit tests exist for all 6 services: `categories.service.spec.ts`, `transactions.service.spec.ts`, `input.service.spec.ts`, `insights.service.spec.ts`, `account.service.spec.ts`, `rate-limit.guard.spec.ts`
2. Unit tests exist for utilities: `parseAmount` (Vietnamese regex), `lookupMerchant` (keyword matching), prompt formatting
3. E2E test at `test/app.e2e-spec.ts` covers all 10 endpoints with happy-path scenarios (not the current `GET /` stub)
4. GitHub Actions workflow runs `lint` → `test` → `migration check` on PR; FAILED status blocks merge

**UI hint:** no

## Phase 5: Performance Optimization

**Goal:** Prevent Vercel timeout kills and memory exhaustion by adding pagination, pushing computation to the database, and fixing inefficient query patterns.

**Requirements:** PERF-01, PERF-02, PERF-03, PERF-04

**Success criteria:**
1. `GET /api/transactions?page=1&limit=50` returns `{ data: [...], total: 243, hasMore: true }` — not all rows
2. `GET /api/insights` computes totals and category breakdowns via SQL `GROUP BY` and `SUM` — no in-memory `reduce()`
3. Month filtering uses `gte(createdAt, '2026-04-01')` / `lt(createdAt, '2026-05-01')` — no `LIKE 'YYYY-MM%'`
4. `vercel.json` has `"maxDuration": 30` — LLM image parsing completes without timeout kill
5. `vercel.json` has `"runtime": "nodejs20.x"` — matches `package.json` engines

**UI hint:** no

## Phase 6: Observability & Documentation

**Goal:** Add production monitoring, structured health checks, API documentation, and deployment hardening — ensuring the backend is observable and well-documented before multi-user launch.

**Requirements:** OBS-01, OBS-02, OBS-03, OBS-04

**Success criteria:**
1. Sentry captures all unhandled NestJS exceptions — visible in Sentry dashboard with stack traces
2. `GET /api/health` returns `{ status: 'ok', database: 'connected', redis: 'connected' }` — verifies actual dependencies
3. Swagger UI available at `/api/docs` — all 10 endpoints documented with request/response schemas
4. `vercel.json` explicitly sets `"runtime": "nodejs20.x"` in build configuration

**UI hint:** no

## Phase Dependencies

```
Phase 1 (Database) ────┬──→ Phase 2 (Security) [independent]
                       │
                       ├──→ Phase 3 (Test Infra) ──→ Phase 4 (Quality Gates)
                       │                                    │
                       ├──→ Phase 5 (Performance) ←─────────┘
                       │
                       └──→ Phase 6 (Observability)

Phase 7 (Differentiators) ← depends on Phase 5 (caching targets post-optimization)
```

- **Phase 1 must be first** — migrations are the foundation for all schema changes
- **Phase 3 depends on Phase 1** — tests need the migration-baseline database state
- **Phase 4 depends on Phase 3** — tests can't be written without test infrastructure
- **Phase 5 benefits from Phase 4** — performance refactors are safer with test coverage
- **Phase 2 is independent** — can run in parallel with Phase 1 or after
- **Phase 6 is independent** — can run anytime, but best after security is fixed
