```typescript
async function bootstrap(): Promise<Express> {
  // Auto-migrate database on cold start (Vercel serverless + local dev).
  // Migrations are idempotent — drizzle-kit tracks applied versions in the database,
  // so re-running on warm instances is a no-op.
  try {
    console.log('[migration] Checking database migrations...');
    await migrate(db, { migrationsFolder: './drizzle' });
    console.log('[migration] Database migrations up to date.');
  } catch (error) {
    console.error('[migration] Migration failed:', error);
    // On Vercel, exit so the function fails fast and the error is visible in logs.
    // On local dev, the developer sees the error and can fix it.
    process.exit(1);
  }

  const app = await NestFactory.create(AppModule, {
```

**Important considerations:**
- The `migrationsFolder: './drizzle'` path is relative to the project root (where `package.json` lives) — this is correct for both Vercel and local dev since both run from the project root.
- `migrate()` is idempotent: drizzle-orm tracks applied migrations in a `__drizzle_migrations` table. On warm Vercel instances with an already-migrated database, the call returns immediately with zero new migrations applied.
- The migration runs BEFORE NestJS creates the app — this ensures all database tables exist before any service constructor or module initializer tries to query them.
- On local dev with `file:local.db`, migrations get applied to the local SQLite file automatically on every `npm run start:dev` restart.

Create `.github/workflows/db-check.yml` with a GitHub Actions workflow that validates migration consistency on every push and PR. Create the `.github/workflows/` directory if it doesn't exist.

```yaml
name: Database Migration Check

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  migration-check:
    name: Check Migration Drift
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Check for schema drift
        run: npx drizzle-kit check
```

**Explanation:**
- `drizzle-kit check` compares the current `src/db/schema.ts` against the latest migration snapshot in `drizzle/meta/`. If any schema change exists without a corresponding migration file, it exits with non-zero status — failing the CI build.
- The workflow runs on every push to main/develop and every PR targeting those branches.
- Node 20 is specified (matches `package.json` engines requirement).
- `npm ci` ensures clean, reproducible installs in CI.

**Note:** This workflow only checks for drift — it does NOT run migrations against any database. Migrations are applied at runtime in main.ts (Task 2). The CI gate catches the common mistake of editing schema.ts without generating a migration.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | drizzle/* SQL files | mitigate | Migration files committed to git — any unauthorized modification is detectable via git history and code review. CI check validates consistency between schema.ts and migrations. |
| T-03-02 | Denial of Service | main.ts auto-migrate | accept | migrate() is idempotent — on warm instances, it's a no-op. Cold start migration overhead is negligible for SQLite (6 tables, ~200 lines of DDL). |
| T-03-03 | Elevation of Privilege | CI workflow | mitigate | CI workflow runs with read-only GitHub token — only reads code, runs drizzle-kit check (no database access, no write permissions). No secrets injected. |
| T-03-04 | Information Disclosure | main.ts console.log | accept | Migration logging outputs table counts, not data. No PII, no credentials in migration output. Acceptable for serverless function logs. |
