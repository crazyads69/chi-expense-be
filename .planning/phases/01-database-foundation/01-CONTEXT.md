# Phase 1: Database Foundation - Context

**Gathered:** 2026-04-26
**Status:** Ready for planning

## Phase Boundary

Establish version-controlled database migrations using drizzle-kit, fix schema inconsistencies, add missing database indices, and fix the UNIQUE constraint handling in the categories service. Switch to HTTP-based Turso client for serverless compatibility. This is pure infrastructure — no API behavior changes.

## Implementation Decisions

### Migration baseline approach
- **D-01:** Generate initial migration via `drizzle-kit generate --name=init` from current `src/db/schema.ts`. This produces a clean SQL migration file that represents the current schema state. No snapshot/migration from existing database needed — the schema file IS the source of truth.
- **D-02:** Commit the generated migration SQL files to version control in `drizzle/` directory. Every future schema change gets its own migration file.
- **D-03:** Migration check (`drizzle-kit check`) added to CI workflow to prevent schema drift.

### Timestamp consistency
- **D-04:** Defer the TEXT→INTEGER timestamp migration for `transactions.created_at` and `categories.created_at` to v1.1. Research (PITFALLS.md) flagged this as high-risk, low-reward — the `LIKE 'YYYY-MM%'` pattern works correctly on ISO text strings, and the multi-step DDL migration required could silently lose data.
- **D-05:** Add a code comment at `src/db/schema.ts:108` and `:127` noting the intentional inconsistency and linking to the deferred migration issue. This prevents confusion for future developers.
- **D-06:** For `PERF-03` (Phase 5), implement date range filtering using the ISO text format (e.g., `>= '2026-04-01' AND < '2026-05-01'`) — this works regardless of column type.

### Missing index
- **D-07:** Add `index('idx_transactions_category').on(table.category)` to the transactions table definition. This is a single extra callback in the existing `sqliteTable` second argument. No migration complexity — drizzle-kit will generate the `CREATE INDEX` statement.

### UNIQUE constraint fix
- **D-08:** Replace error string matching in `src/categories/categories.service.ts:52-63` with Drizzle's `.onConflictDoNothing()` method on the insert query. This eliminates the fragile `error.message.includes('UNIQUE constraint failed')` check.
- **D-09:** Remove the recursive `this.list(userId)` call at line 60. After `.onConflictDoNothing()`, simply re-query categories (single SELECT) instead of recursing into the full `list()` method which could re-attempt the INSERT.

### Client protocol
- **D-10:** Switch `src/db/client.ts` from default `@libsql/client` (WebSocket transport) to `@libsql/client/http` for stateless serverless compatibility on Vercel. The HTTP client uses REST API calls instead of persistent WebSocket connections.
- **D-11:** The `TURSO_SYNC_URL` env var becomes unused with HTTP client (no embedded replica sync). Keep it in `.env.example` for future use, but document that HTTP client doesn't use it.

### Migration deployment
- **D-12:** Auto-migrate on Vercel cold start in the `bootstrap()` function in `src/main.ts`. Before creating the NestJS app, check if migrations need applying. On local dev, `npm run start:dev` will also auto-migrate.
- **D-13:** Add a `db:migrate` npm script for manual migration application if needed: `drizzle-kit migrate`.

### the agent's Discretion
- Exact migration SQL content (generated by drizzle-kit)
- Whether to create a `db/migrate.ts` helper or inline the migration code in `main.ts`
- Error handling for failed migrations (exit process with clear message)
- Timestamp comment wording in schema.ts

## Specific Ideas

- Migrations must be idempotent — running them multiple times on the same database should be safe
- The migration workflow should work on both local `local.db` (SQLite file) and remote Turso databases
- CI should catch migration drift early — if someone changes schema.ts without generating a migration, the build fails

## Existing Code Insights

### Reusable Assets
- `src/db/schema.ts` — Complete schema with 6 tables, 3 existing indices. The second callback argument pattern `(table) => [index(...), index(...)]` is established and will be followed for adding the new category index.
- `src/db/client.ts` — Simple 11-line file. The `createClient()` call at module level will be switched from WebSocket to HTTP transport.
- `drizzle.config.ts` — Already properly configured with `dialect: 'turso'`, `schema: './src/db/schema.ts'`, and `out: './drizzle'`. No config changes needed.

### Established Patterns
- Index definition pattern: All indices are defined in the second callback of `sqliteTable()`, not as separate statements. The category index follows this.
- Timestamp pattern split: Better Auth tables use `integer({ mode: 'timestamp_ms' })` with SQL defaults. App tables use `text().notNull()` with JS `new Date().toISOString()`. This inconsistency is accepted for v1.0.
- Foreign key cascade: All child tables use `.references(() => user.id, { onDelete: 'cascade' })`. No changes needed.

### Integration Points
- `src/main.ts` — Bootstrap function where migration auto-apply will be added
- `src/categories/categories.service.ts:50-63` — Try/catch block to replace with `.onConflictDoNothing()`
- `src/app.module.ts` — No changes needed for Phase 1 (ConfigModule already loads env vars needed by db client)
- GitHub Actions CI (new) — Migration check step will be added

## Deferred Ideas

- Timestamp TEXT→INTEGER migration for transactions and categories — deferred to v1.1 (high risk, low reward for v1.0)
- Turso embedded replica sync (TURSO_SYNC_URL) — not compatible with HTTP client; revisit if sync features needed
- Connection pooling optimization — deferred until load testing reveals actual bottlenecks
- Database backup/restore automation — separate phase or v1.1 enhancement
