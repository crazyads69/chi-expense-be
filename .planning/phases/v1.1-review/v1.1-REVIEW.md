# Code Review: v1.1 Changes

**Reviewed:** 2026-04-27
**Depth:** standard
**Files Reviewed:** 27
**Status:** issues_found

## Summary

This review covers the complete v1.1 milestone changes across phases 8–13 (Request Context, API Versioning, Apple OAuth, Caching/Tracing, Timeouts/Image Optimization, Reliability/DX). The codebase shows solid NestJS patterns, good use of Drizzle ORM for safe SQL, and thoughtful DX additions like Swagger docs and health checks. However, one **critical data-integrity bug** in the categories service corrupts API responses by returning slugs as IDs. Additionally, there are several warnings around input validation gaps, production security posture, error-handling semantics, and code quality issues that should be addressed before shipping.

---

## Critical Issues

### CR-01: Categories API returns slug as category `id`, corrupting the API contract

- **Severity:** critical
- **File:** `src/categories/categories.service.ts`
- **Lines:** 100, 108
- **Issue:** In both the lazy-initialization path (line 100) and the existing-categories path (line 108), the service maps `id: cat.slug` instead of `id: cat.id`. The database stores a unique `nanoid()` primary key in `cat.id`, but callers receive the human-readable slug in the `id` field. The `CategoryResponse` interface declares separate `id` and `slug` fields, so this violates the API contract. Any downstream feature that stores these IDs and later uses them to reference categories (e.g., updates, filtering, foreign-key-style lookups) will fail because the slug does not match the real primary key.
- **Recommendation:** Change both `map` blocks to return `id: cat.id`.

```typescript
// Before (incorrect)
return initializedCategories.map((cat) => ({
  name: cat.name,
  slug: cat.slug,
  id: cat.slug,        // ← BUG: should be cat.id
  budget: cat.budget,
}));

// After (correct)
return initializedCategories.map((cat) => ({
  name: cat.name,
  slug: cat.slug,
  id: cat.id,
  budget: cat.budget,
}));
```

## Warnings

### WR-01: `parseMonth` regex allows invalid months, producing silently wrong date boundaries

- **Severity:** warning
- **File:** `src/lib/date-utils.ts`
- **Lines:** 12
- **Issue:** The regex `/^\d{4}-\d{2}$/` accepts strings like `2026-13` or `2026-00`. `getMonthBoundaries` then passes these to `new Date('2026-13-01T00:00:00.000Z')`, which in JavaScript **wraps** to a different month (e.g., `2026-13` becomes February 2027). The function returns incorrect boundaries without throwing, so the insights/transactions APIs silently return data for the wrong month.
- **Recommendation:** Validate the month is `01`–`12`:

```typescript
if (!/^\d{4}-(0[1-9]|1[0-2])$/.test(month)) {
  throw new Error('Invalid month format. Expected YYYY-MM');
}
```

### WR-02: Query pagination params parsed with `parseInt` but not validated, allowing `NaN` to propagate

- **Severity:** warning
- **File:** `src/transactions/transactions.controller.ts`
- **Lines:** 35–36
- **Issue:** `page` and `limit` query strings are passed through `parseInt` without `isNaN` checks. If a client sends `?page=abc`, `pageNum` becomes `NaN`. In the service, `Math.max(1, NaN)` yields `NaN`, and `offset = NaN * 50` is also `NaN`. Drizzle may throw an opaque error or ignore the clause, resulting in unpredictable pagination behavior.
- **Recommendation:** Validate after parsing, or use a `ParseIntPipe` / `DefaultValuePipe` from `@nestjs/common`:

```typescript
async list(
  @Session() session: UserSession,
  @Query('month') month?: string,
  @Query('page', new DefaultValuePipe(1), ParseIntPipe) page?: number,
  @Query('limit', new DefaultValuePipe(50), ParseIntPipe) limit?: number,
) { ... }
```

### WR-03: `parseAmount` regex matches `ng` prefix in common Vietnamese words, causing false positives

- **Severity:** warning
- **File:** `src/input/input.service.ts`
- **Lines:** 48
- **Issue:** The first regex pattern includes `ng` as a thousands shorthand (`nghìn`). Because the pattern is case-insensitive and `ng` is not a word boundary, inputs like `"đi 3 ngày"` match as `3 ng`, and the parser returns `3000` instead of `3` (or no amount). This affects the local-parse fallback path used when merchant keywords match.
- **Recommendation:** Use word-boundary assertions or require `ng` to be followed by a word boundary or the full `nghìn`:

```typescript
/(\d{1,3}(?:,\d{3})*(?:[.,]\d+)?)\s*(?:k|nghìn)\b/i
```

### WR-04: `bodyParser: false` may break non-auth POST endpoints if better-auth does not parse all routes

- **Severity:** warning
- **File:** `src/main.ts`
- **Lines:** 35
- **Issue:** `NestFactory.create(AppModule, { bodyParser: false })` disables Express's default body parser. The comment states this is required by `@thallesp/nestjs-better-auth`, but there is no visible middleware in `main.ts` that re-enables JSON body parsing for non-auth routes (e.g., `POST /api/transactions`, `POST /api/input/text`). If the better-auth module only parses its own routes, other POST endpoints will receive `req.body === undefined`, causing `ValidationPipe` to reject requests or throw unexpected errors.
- **Recommendation:** Verify that all POST routes receive parsed bodies under this configuration. If not, register `express.json()` manually after better-auth middleware:

```typescript
import { json } from 'express';
// After app creation but before global pipes
app.use(json());
```

### WR-05: Swagger UI exposed unconditionally in production

- **Severity:** warning
- **File:** `src/main.ts`
- **Lines:** 99–104
- **Issue:** The Swagger documentation endpoint (`/api/docs`) is registered without an environment guard. In production, this exposes the full API schema, authentication mechanisms, and endpoint details to potential attackers, expanding the attack surface for reconnaissance.
- **Recommendation:** Gate Swagger setup behind an environment check, or protect it with basic auth:

```typescript
if (process.env.NODE_ENV !== 'production') {
  const document = SwaggerModule.createDocument(app, swaggerConfig);
  SwaggerModule.setup('api/docs', app, document, { ... });
}
```

### WR-06: CORS origin fallback allows `localhost:8081` in production

- **Severity:** warning
- **File:** `src/main.ts`
- **Lines:** 63, 72
- **Issue:** `allowedOrigins` falls back to `http://localhost:8081` when `FRONTEND_URL` is unset. In a production deployment where `FRONTEND_URL` is accidentally omitted, browsers from any origin could potentially exploit the localhost fallback (though modern browsers block this for localhost). More critically, the origin check allows `!origin` (no origin header), which is correct for mobile apps but means the origin whitelist is not strictly enforced for non-browser clients.
- **Recommendation:** Fail fast at startup if `FRONTEND_URL` is missing in production, or require an explicit `ALLOWED_ORIGINS` list.

### WR-07: Health check returns HTTP 200 when critical dependencies are down

- **Severity:** warning
- **File:** `src/health.controller.ts`
- **Lines:** 42–74
- **Issue:** The `/api/health` endpoint returns `{ status: 'degraded' }` with HTTP 200 when the database or Redis is unreachable. If a load balancer or Kubernetes uses this endpoint as a readiness probe, it will continue routing traffic to an instance that cannot serve requests, causing cascading failures.
- **Recommendation:** Return HTTP 503 when any critical dependency is unhealthy:

```typescript
if (!allHealthy) {
  throw new ServiceUnavailableException({ status: 'degraded', database, redis, timestamp });
}
```

### WR-08: Category lazy-init is not wrapped in a database transaction

- **Severity:** warning
- **File:** `src/categories/categories.service.ts`
- **Lines:** 69–111
- **Issue:** The `fetchFromDb` method performs a SELECT, conditional INSERT, and second SELECT outside of a transaction. Although the unique index on `(userId, slug)` prevents duplicate rows from concurrent INSERTs, the non-transactional read-then-write pattern is brittle. If the unique constraint were ever removed or renamed, concurrent requests would create duplicate categories. Additionally, the cache write (line 61) happens outside the DB transaction, so a DB rollback could leave stale cache data.
- **Recommendation:** Wrap the select-check-insert-select logic in `this.db.transaction()` and invalidate/update the cache only after the transaction commits.

### WR-09: Insights service executes three separate DB queries where one or two would suffice

- **Severity:** warning
- **File:** `src/insights/insights.service.ts`
- **Lines:** 50–77
- **Issue:** `getMonthlyInsights` runs three round-trips to the database: one for totals, one for category breakdown, and one for daily expenses. All three filter on the same `whereClause`. At scale, this multiplies query latency and connection pool pressure unnecessarily. The totals can be derived client-side from the category breakdown, and the daily expenses could be combined or fetched in a single query with window functions.
- **Recommendation:** Compute `total` and `count` from the `categoryBreakdown` array in memory, eliminating the first query. Evaluate combining the remaining two queries if the ORM supports it.

### WR-10: Request-context interceptor leaks inner Observable subscription on unsubscribe

- **Severity:** warning
- **File:** `src/lib/request-context.interceptor.ts`
- **Lines:** 27–35
- **Issue:** The interceptor returns a new `Observable` that manually bridges `next.handle().subscribe()` to an outer subscriber. It does not return a teardown function, so if the HTTP request is aborted and the outer subscriber unsubscribes, the inner subscription to `next.handle()` continues to exist until it completes, leaking memory and potentially running handlers for abandoned requests.
- **Recommendation:** Capture the inner subscription and return an unsubscribe handler:

```typescript
return new Observable((subscriber) => {
  const subscription = requestContext.run({ requestId, startTime }, () => {
    return next.handle().subscribe({
      next: (value) => subscriber.next(value),
      error: (err) => subscriber.error(err),
      complete: () => subscriber.complete(),
    });
  });
  return () => subscription.unsubscribe();
});
```

### WR-11: `userId` parameter accepted but never used in text/image parsing services

- **Severity:** warning
- **File:** `src/input/input.service.ts`
- **Lines:** 96, 189
- **Issue:** Both `parseText(userId, message)` and `parseImage(userId, imageBase64)` receive `userId` but never reference it. This means no user-scoped logging, no per-user Sentry tags, and no audit trail linking parsed expenses to the requesting user within the service layer. If the guard or controller ever passes the wrong identifier, the error would be undetectable.
- **Recommendation:** Use `userId` for Sentry span tags or structured logging:

```typescript
this.logger.log({ userId, message }, 'Parsing text expense');
Sentry.setTag('user_id', userId);
```

### WR-12: Redis client lazily initializes with empty credentials instead of failing fast

- **Severity:** warning
- **File:** `src/lib/redis.ts`
- **Lines:** 16–19
- **Issue:** `getRedisClient` falls back to empty strings for `url` and `token` if environment variables are missing. The error is deferred until the first Redis operation, making production misconfiguration harder to diagnose at startup. Additionally, `redisClient` is declared at module scope without type safety (`let redisClient: Redis` is fine, but no explicit undefined initial state).
- **Recommendation:** Validate required Redis env vars in `app.module.ts` validation (already done for `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN`), but also fail explicitly in `getRedisClient` if they are empty:

```typescript
const url = process.env.UPSTASH_REDIS_REST_URL;
const token = process.env.UPSTASH_REDIS_REST_TOKEN;
if (!url || !token) {
  throw new Error('Redis configuration missing: UPSTASH_REDIS_REST_URL and UPSTASH_REDIS_REST_TOKEN are required');
}
```

### WR-13: `updateData` typed as `Record<string, any>` loses type safety in transaction update

- **Severity:** warning
- **File:** `src/transactions/transactions.service.ts`
- **Lines:** 103
- **Issue:** The `update` method spreads the DTO into a `Record<string, any>` and conditionally mutates `amount`. This bypasses TypeScript checking for the Drizzle `set()` call. If the DTO ever gains a field not present in the schema, or if a typo is introduced, the error will only surface at runtime.
- **Recommendation:** Build a typed partial object explicitly:

```typescript
const updateData: Partial<typeof transactions.$inferInsert> = {
  ...dto,
  updatedAt: nowISO(),
};
if (dto.amount !== undefined) {
  updateData.amount = -Math.abs(dto.amount);
}
```

### IN-06: New OpenAI client instantiated on every LLM call

- **Severity:** info
- **File:** `src/lib/openrouter.ts`
- **Lines:** 5–12
- **Issue:** `getOpenAIClient()` creates a fresh `OpenAI` instance per request. Depending on the SDK version, this may create a new HTTP agent each time, increasing connection overhead.
- **Recommendation:** Cache a single client instance:

```typescript
let client: OpenAI | undefined;
export const getOpenAIClient = () => {
  if (!client) {
    client = new OpenAI({ ... });
  }
  return client;
};
```

### IN-07: Sentry initialized with empty DSN string when env var is missing

- **Severity:** info
- **File:** `src/instrument.ts`
- **Lines:** 3–8
- **Issue:** `dsn: process.env.SENTRY_DSN || ''` passes an empty string to `Sentry.init()`. Sentry may log a warning or attempt to initialize anyway. `tracesSampleRate` and `profilesSampleRate` are also hardcoded.
- **Recommendation:** Conditionally initialize and make sample rates configurable:

```typescript
if (process.env.SENTRY_DSN) {
  Sentry.init({
    dsn: process.env.SENTRY_DSN,
    tracesSampleRate: parseFloat(process.env.SENTRY_TRACES_SAMPLE_RATE || '0.1'),
  });
}
```

### IN-08: Categories cache has no invalidation mechanism

- **Severity:** info
- **File:** `src/categories/categories.service.ts`
- **Lines:** 20–66
- **Issue:** Once categories are cached in Redis (60s TTL), there is no way to invalidate the cache if categories are modified. Currently there are no modification endpoints, but when they are added, stale cache data will be served.
- **Recommendation:** Add a `invalidateCategoriesCache(userId)` helper and call it from any future create/update/delete category methods.
